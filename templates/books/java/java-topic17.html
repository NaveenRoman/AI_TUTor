Conflict demo (brief)

Typical conflict scenario: two branches modify same lines. Resolve by editing file(s), test locally, then git add and git commit. Use git merge --abort to cancel if needed.

Pro tip: rebase feature branch onto latest dev before opening PR to keep history linear: git fetch origin && git rebase origin/dev

Pull Request checklist (for reviewers)
  • Does code compile and tests pass? (mvn -DskipTests=false test)
  • Are changes small and focused?
  • Is there documentation or a README update if required?
  • Security & performance impact considered?
Tools: GitHub, GitLab, Bitbucket, GitHub CLI (gh), SourceTree, GitKraken, VSCode built-in Git.

3Ô∏è‚É£ Unit Testing ‚Äî JUnit 5 (Beginner ‚Üí Advanced)

Testing

Write deterministic, fast unit tests. Use parameterized tests, lifecycle methods, and mocks for dependencies.

Basic test

import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;

class MathTest {
  @Test
  void add() { assertEquals(5, 2 + 3); }
}

Parameterized test

import org.junit.jupiter.params.*;
import org.junit.jupiter.params.provider.*;
class ParamTest {
  @ParameterizedTest
  @CsvSource({ "2,3,5", "1,1,2" })
  void add(int a,int b,int expected){ assertEquals(expected, a+b); }
}
Test lifecycle & mocking
  • @BeforeEach / @AfterEach for per-test setup/cleanup.
  • @BeforeAll / @AfterAll for expensive shared setup (static).
  • Use Mockito or MockK to mock external dependencies (DB, web clients).
CI integration (quick)

In CI (GitHub Actions / Jenkins) run mvn -DskipTests=false test or ./gradlew test. Fail build on test failures and integrate code coverage (JaCoCo).

4Ô∏è‚É£ Logging ‚Äî SLF4J & Logback (best practices)

Observability

Use SLF4J as API and Logback (or Log4j2) as implementation. Prefer structured logs and include context (request-id, user-id).

Code (SLF4J)

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class LogDemo {
  private static final Logger log = LoggerFactory.getLogger(LogDemo.class);
  public static void main(String[] args){
    log.info("Application started");
    try{ int x=1/0; } catch(Exception e){ log.error("Processing failed: {}", e.toString()); }
  }
}

logback.xml (simple)

<configuration>
  <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
    <encoder><pattern>%d{yyyy-MM-dd HH:mm:ss} %-5level [%thread] %logger{36} - %msg%n</pattern></encoder>
  </appender>
  <root level="INFO"><appender-ref ref="STDOUT"/></root>
</configuration>
Logging best practices
  • Don't log sensitive data (PII, passwords).
  • Use parameterized logging to avoid unnecessary String concatenation.
  • Use correlation IDs (MDC) for tracing across services.

5Ô∏è‚É£ Spring Boot ‚Äî DI, REST API, Config

Framework

Spring Boot accelerates building production-ready apps: Auto configuration, embedded server, and opinionated defaults. Focus: Controller ‚Üí Service ‚Üí Repository (DI).

Layer diagram 
Client (HTTP) ‚Üí Controller ‚Üí Service ‚Üí Repository ‚Üí Database

Main application

@SpringBootApplication
public class DemoApp {
  public static void main(String[] args){ SpringApplication.run(DemoApp.class, args); }
}

Controller + Service (example)

@RestController
@RequestMapping("/api")
public class UserController {
  private final UserService userService;
  public UserController(UserService s){ this.userService = s; }
  @GetMapping("/hello") public String hello(){ return userService.greet(); }
}

@Service
class UserService {
  public String greet(){ return "Hello from Spring Boot!"; }
}
application.properties example 
server.port=8080

Why DI matters (concise)

Dependency Injection decouples object creation from usage, making code testable and modular. In Spring, beans are managed by the container, injected via constructors (preferred) or @Autowired.

Pro tips for Spring interviews
  • Be ready to explain @Component vs @Service vs @Repository and bean scopes.
  • Describe how Spring Boot autoconfiguration works (starters, conditional beans).
  • Know about transactions (@Transactional), caching, and actuator endpoints for monitoring.

CI/CD pipeline overview (GitHub Actions example)

Typical flow: push ‚Üí CI (build, test) ‚Üí build artifact ‚Üí push to registry ‚Üí deploy to staging ‚Üí smoke tests ‚Üí deploy production.

name: Java CI
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with: java-version: '17'
      - name: Build with Maven
        run: mvn -B -DskipTests=false package
      - name: Publish artifact
        uses: actions/upload-artifact@v3
        with: name: app-jar; path: target/*.jar
Cloud quick notes
  • Deploy JARs to AWS Elastic Beanstalk, ECS (Docker), or serverless containers (AWS Fargate).
  • Use environment variables for secrets (do NOT commit creds).
  • Integrate monitoring (Prometheus + Grafana) and logs (ELK/EFK stack).

Pro Tips & Interview Pointers

  • Know not only commands, but why: explain what happens under the hood (e.g., git objects, refs, staging area).
  • Show CI knowledge: how to add a test step and fail fast on regressions.
  • In interviews, explain trade-offs: fat JAR vs thin JAR + dependencies, container image size vs startup time.
  • For production: use health checks, graceful shutdown, and rolling updates for zero-downtime deploys.
