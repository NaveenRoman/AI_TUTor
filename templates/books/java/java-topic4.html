üß± 4. Object-Oriented Programming (OOP)

Java is a pure Object-Oriented Programming language. Everything revolves around **classes** and **objects** that model real-world entities.

‚úÖ Classes, Objects, and Members

Definition

A **Class** is a logical blueprint or template defining the structure and behavior of entities. An **Object** is a run-time entity, a specific instance of a class that occupies memory.

5 Key Points

  • A Class does not consume memory; an Object consumes memory upon creation (instantiation).
  • Class Members are **fields** (data/state) and **methods** (behavior).
  • Objects are created using the `new` keyword, which allocates memory on the heap.
  • Each object has its own unique set of non-static instance variables.
  • **Real-Life Use Case:** The Class is the **design plan for a house**, and the Objects are the **actual, physical houses** built from that plan.

Example Program

// Line 1: Define the Blueprint (Class)
class Dog {
  // Line 2: Field/State (Instance variable)
  String breed; 
  // Line 3: Method/Behavior
  void bark() {
    System.out.println(breed + " says Woof!");
  }
}

public class Main {
  public static void main(String[] args) {
    // Line 4: Create Object 1 (Instance)
    Dog myDog = new Dog(); 
    // Line 5: Set state
    myDog.breed = "Golden Retriever"; 
    // Line 6: Call behavior
    myDog.bark(); 
  }
}

Line-by-Line Explanation

  • `class Dog {` : Starts the blueprint definition.
  • `String breed;` : Defines a piece of state unique to every `Dog` object.
  • `void bark() { ... }` : Defines the behavior that all `Dog` objects can perform.
  • `Dog myDog = new Dog();` : Instantiates the class, creating an object named `myDog` in memory.
  • `myDog.breed = "Golden Retriever";` : Accesses the `breed` field of the specific `myDog` object and sets its value.
  • `myDog.bark();` : Invokes the `bark` method on the `myDog` object.

‚úÖ Constructors (types, overloading)

Definition

A **constructor** is a special type of method used to initialize an object when it is created. It ensures that the object starts in a valid, ready-to-use state.

5 Key Points

  • It must have the **exact same name** as its class.
  • It has **no explicit return type** (not even `void`).
  • If no constructor is defined, Java provides a **default constructor**.
  • Constructors can be **overloaded** (multiple constructors with different parameters).
  • **Real-Life Use Case:** When you sign up for a service, a constructor initializes your **User object** with mandatory fields like `userID` and the current `timestamp`.

Example Program

// Line 1: Define the Class
class Account {
  String type;
  double balance;

  // Line 2: Default Constructor (Initializes to default state)
  Account() {
    this.type = "Savings";
    this.balance = 0.0;
  }

  // Line 3: Parameterized Constructor (Allows custom initialization)
  Account(String type, double initialBalance) {
    this.type = type;
    this.balance = initialBalance;
  }

  // Line 4: Displays account state
  void showDetails() {
    System.out.println(this.type + " Account, Balance: $" + this.balance);
  }
}

public class Main {
  public static void main(String[] args) {
    // Line 5: Use Default Constructor
    Account acc1 = new Account(); 
    // Line 6: Use Parameterized Constructor
    Account acc2 = new Account("Checking", 500.75); 
    acc1.showDetails();
    acc2.showDetails();
  }
}

Line-by-Line Explanation

  • `class Account { ... }` : Defines the account blueprint.
  • `Account() { ... }` : The default constructor, providing basic initialization for `acc1`.
  • `Account(String type, double initialBalance) { ... }` : The parameterized constructor, allowing the user to specify initial values for `acc2`.
  • `void showDetails() { ... }` : A simple method to print the object's current state.
  • `Account acc1 = new Account();` : Creates an object using the default (no-argument) constructor.
  • `Account acc2 = new Account("Checking", 500.75);` : Creates an object using the two-argument constructor.

‚úÖ this, super, and static keywords

Definition

These are utility keywords used for referencing, scope management, and memory control within classes and objects.

5 Key Points

  • `this` is a reference to the **current object** (instance) of the method or constructor.
  • `super` is a reference to the **immediate parent class** object.
  • `static` members belong to the **class** rather than any specific object.
  • `static` methods can only access other `static` members; they cannot use `this` or `super`.
  • **Real-Life Use Case:** `static` is used extensively in **Utility classes** like `java.lang.Math` (e.g., `Math.sqrt(x)`). You never create a `Math` object; you use the class directly.

Example Program

// Line 1: Parent Class
class Parent {
  int x = 10;
  Parent(int x) { this.x = x; } // 'this' resolves ambiguity
}

// Line 2: Child Class
class Child extends Parent {
  int x = 20;
  // Line 3: Static variable, shared across all Child objects
  static int count = 0; 

  Child() {
    // Line 4: Call parent constructor
    super(5); 
    Child.count++; 
  }

  void display() {
    // Line 5: Access current object's field
    System.out.println("Child x: " + this.x); 
    // Line 6: Access parent's field (hidden by Child's x)
    System.out.println("Parent x: " + super.x); 
  }
}

public class Main {
  public static void main(String[] args) {
    new Child().display();
    new Child();
    // Line 7: Access static member via class name
    System.out.println("Total Children: " + Child.count); 
  }
}

Line-by-Line Explanation

  • `class Parent { ... }` : Defines the base class with field `x` and a constructor using `this` to assign the parameter value to the instance variable.
  • `class Child extends Parent { ... }` : Defines the subclass, introducing its own `x` and the shared `count`.
  • `static int count = 0;` : Declares a field shared by all `Child` objects; tracks how many objects were created.
  • `super(5);` : Must be the first line in the constructor, calling the matching parameterized constructor in the `Parent` class.
  • `System.out.println("Child x: " + this.x);` : `this.x` refers to the `x` variable declared in the `Child` class (value 20).
  • `System.out.println("Parent x: " + super.x);` : `super.x` refers to the `x` variable inherited from the `Parent` class (value 5, set by `super(5)`).
  • `System.out.println("Total Children: " + Child.count);` : Accesses the `static` variable directly using the class name, without needing an object.

üîí Access Modifiers (public, private, protected, default)

Definition

Access modifiers control the **visibility** and **scope** of a class, field, method, or constructor, acting as a gatekeeper for access from other classes or packages.

5 Key Points

  • **`public`**: Accessible from anywhere in the Java application.
  • **`private`**: Accessible only within its own class (key to Encapsulation).
  • **`protected`**: Accessible within the same package AND by subclasses in any package.
  • **Default (no keyword)**: Accessible only within the same package.
  • **Real-Life Use Case:** Your personal bank account **PIN** is `private`, while the **ATM machine** itself is `public`.

üîê Encapsulation (Getters/Setters)

Definition

Encapsulation is the mechanism of **bundling data (fields) and the code that operates on the data (methods)** into a single unit (the class), and restricting direct access to the data (often using the `private` keyword).

5 Key Points

  • Achieved primarily by making instance variables `private`.
  • Public methods (Getters/Setters) are provided to read and modify the private data.
  • The internal data representation can change without affecting outside code.
  • Allows for **data validation** and security checks inside Setter methods.
  • **Real-Life Use Case:** A **Car's Gear System**. The driver interacts with a public lever (the Setter method), but the complex gears and clutches inside the transmission (the private data) are hidden and protected.

Example Program

// Line 1: Class with private data
class UserProfile {
  private int age; 

  // Line 2: Public Setter with validation logic
  public void setAge(int age) {
    if (age > 0 && age < 120) {
      this.age = age; // Data is set only if valid
    } else {
      System.out.println("Invalid age provided.");
    }
  }

  // Line 3: Public Getter
  public int getAge() {
    return age; 
  }
}

public class Main {
  public static void main(String[] args) {
    UserProfile user = new UserProfile();
    user.setAge(25); // Valid
    user.setAge(-5);  // Invalid, rejected by Setter logic
    System.out.println("Current Age: " + user.getAge()); 
  }
}

Line-by-Line Explanation

  • `private int age;` : Hides the data from direct external modification.
  • `public void setAge(int age) { ... }` : The control point. It validates the input (`age > 0 && age < 120`). If the validation fails, the internal `age` field is not updated.
  • `public int getAge() { ... }` : Allows read-only access to the private data.

üß¨ Inheritance (Single, Multilevel, Hierarchical)

Definition

Inheritance is the process where one class (the **Child** or subclass) acquires the properties (fields and methods) of another class (the **Parent** or superclass), establishing an **"is-a"** relationship.

5 Key Points

  • Uses the `extends` keyword in the class definition.
  • Promotes **code reusability** by avoiding duplicate code in subclasses.
  • Only **single inheritance** (one parent) is supported for classes in Java.
  • The parent's constructor is implicitly or explicitly called via `super()`.
  • **Real-Life Use Case:** A **Bank Account** system. `SavingsAccount` and `CheckingAccount` both **extend** the base `Account` class to reuse common logic like `deposit()` and `withdraw()`.

Example Program (Multi-level Inheritance)

// Line 1: Grandparent Class
class Device {
  void powerOn() { System.out.println("Device is ON."); }
}

// Line 2: Parent Class (inherits from Device)
class SmartPhone extends Device {
  void call() { System.out.println("Phone calling..."); }
}

// Line 3: Child Class (inherits from SmartPhone)
class AndroidPhone extends SmartPhone {
  void installApp() { System.out.println("Installing Android app."); }
}

public class Main {
  public static void main(String[] args) {
    AndroidPhone myPhone = new AndroidPhone();
    // Line 4: Method from Grandparent
    myPhone.powerOn(); 
    // Line 5: Method from Parent
    myPhone.call(); 
    // Line 6: Method from Child
    myPhone.installApp(); 
  }
}

Line-by-Line Explanation

  • `class Device { ... }` : Base class with the most general behavior.
  • `class SmartPhone extends Device { ... }` : Inherits `powerOn()`.
  • `class AndroidPhone extends SmartPhone { ... }` : Inherits both `powerOn()` and `call()` (Multi-level inheritance).
  • `myPhone.powerOn();` : Accesses the method inherited from `Device`.
  • `myPhone.call();` : Accesses the method inherited from `SmartPhone`.
  • `myPhone.installApp();` : Accesses the method defined in `AndroidPhone`.

üé≠ Polymorphism (Compile & Runtime)

Definition

Polymorphism means **"many forms"**. It is the ability of a method to act differently based on the number/type of its parameters (**Compile-time**) or the actual object type at runtime (**Runtime**).

5 Key Points

  • **Compile-time Polymorphism** is achieved through **Method Overloading**.
  • **Runtime Polymorphism** is achieved through **Method Overriding**.
  • Method Overloading requires different method signatures (parameters).
  • Runtime Polymorphism relies on the JVM deciding which overridden method to execute during execution (Dynamic Method Dispatch).
  • **Real-Life Use Case:** A **Speaker System**. The `play()` method behaves differently depending on the input: `play(CD)` plays music, `play(RadioStation)` tunes in, and `play(BluetoothDevice)` streams audio.

Example Program (Runtime Polymorphism - Overriding)

// Line 1: Parent Class
class Payment {
  void process() { 
    System.out.println("Processing general payment..."); 
  }
}

// Line 2: Subclass 1
class CreditCard extends Payment {
  // Line 3: Overriding the parent method
  @Override 
  void process() { 
    System.out.println("Processing payment via Credit Card."); 
  }
}

// Line 4: Subclass 2
class PayPal extends Payment {
  // Line 5: Overriding the parent method
  @Override 
  void process() { 
    System.out.println("Processing payment via PayPal wallet."); 
  }
}

public class Main {
  public static void main(String[] args) {
    // Line 6: Reference of Parent type, but object of Child type
    Payment p1 = new CreditCard(); 
    Payment p2 = new PayPal();
    
    // Line 7: The same method call executes different logic based on the object type
    p1.process(); 
    p2.process(); 
  }
}

Line-by-Line Explanation

  • `class Payment { ... }` : The base class, defining the contract (`process()`).
  • `class CreditCard extends Payment { ... }` : Subclass inheriting from `Payment`.
  • `@Override void process() { ... }` : Replaces the parent's implementation with credit card-specific logic.
  • `class PayPal extends Payment { ... }` : Another subclass providing its own implementation.
  • `@Override void process() { ... }` : Replaces the parent's implementation with PayPal-specific logic.
  • `Payment p1 = new CreditCard();` : This is the core of Runtime Polymorphism: the reference type is `Payment`, but the object type is `CreditCard`.
  • `p1.process();` : The JVM looks at the **actual object** (`CreditCard`) and executes its overridden `process()` method, not the `Payment` version.

üß± Abstract Class & Interface (Abstraction)

Definition

**Abstraction** is the principle of hiding the complex implementation details and showing only the essential features. It is achieved in Java using **Abstract Classes** (partial abstraction) and **Interfaces** (100% abstraction).

5 Key Points

  • An **Abstract Class** can have both implemented (concrete) methods and unimplemented (`abstract`) methods.
  • An Abstract Class must be defined with the `abstract` keyword and **cannot be instantiated**.
  • An **Interface** only specifies method signatures (contracts); pre-Java 8, it couldn't have method bodies.
  • Classes **extend** abstract classes but **implement** interfaces.
  • **Real-Life Use Case:** The **Abstract Class** could be a `RemoteControl` (defining `powerOn`, `changeChannel`). Different **Interfaces** like `SmartDevice` or `Infrared` could be implemented by the control to handle varying connections.

Example Program

// Line 1: Interface (100% abstract contract)
interface Connectable {
  void connect(); // Abstract and public by default
}

// Line 2: Abstract Class (Partial implementation)
abstract class Device {
  void displayInfo() { // Concrete method
    System.out.println("Displaying device information..."); 
  }
  abstract void turnOff(); // Abstract method (no body)
}

// Line 3: Concrete Class implements Interface and extends Abstract Class
class Laptop extends Device implements Connectable {
  @Override
  public void connect() { // Implements Connectable contract
    System.out.println("Laptop connected to Wi-Fi.");
  }
  @Override
  void turnOff() { // Implements Device contract
    System.out.println("Laptop shutting down.");
  }
}

public class Main {
  public static void main(String[] args) {
    Laptop myLaptop = new Laptop();
    myLaptop.displayInfo();
    myLaptop.connect();
    myLaptop.turnOff();
  }
}

Line-by-Line Explanation

  • `interface Connectable { ... }` : Defines a behavioral contract that any implementing class must fulfill (`connect()`).
  • `abstract class Device { ... }` : Defines a base that provides some common logic (`displayInfo()`) but forces subclasses to define how they turn off (`turnOff()`).
  • `class Laptop extends Device implements Connectable { ... }` : The concrete class that must provide the bodies for `connect()` and `turnOff()`.

üö´ Final, Static, and Abstract Modifiers

These non-access modifiers govern the characteristics of class members or classes themselves.

Final

  • Variable: Makes it a **constant**.
  • Method: Prevents **overriding**.
  • Class: Prevents **inheritance**.

Static

  • Member: Belongs to the **class**, shared by all objects.
  • Block: Executed only once when the class is loaded.
  • Used for utility functions (e.g., `main` method).

Abstract

  • Method: Signature only (no body); forces implementation.
  • Class: Cannot be instantiated; meant only for extension.
  • Enforces **abstraction**.

‚ú® The Object Class: equals(), hashCode(), etc.

Definition

The `java.lang.Object` class is the root of the Java class hierarchy. Every class is a descendant of `Object` and automatically inherits its non-private methods, such as `equals()`, `hashCode()`, and `toString()`.

5 Key Points

  • The default `equals()` checks for **reference equality** (if two variables point to the same memory location).
  • The default `hashCode()` returns a unique integer based on the object's memory address.
  • It is crucial to override both `equals()` and `hashCode()` for **content equality** when using objects in collections (e.g., `HashMap`, `HashSet`).
  • The `toString()` method is used to provide a human-readable representation of an object.
  • **Real-Life Use Case:** When comparing two products in an e-commerce cart. You don't care if they are the exact same physical object in memory; you care if they have the same **Product ID** and **Options**. This requires overriding `equals()`.

Example Program (Custom Equality)

import java.util.Objects;

// Line 1: Define a Product class
public class Product {
  private final int productId;
  private final String name;

  public Product(int productId, String name) {
    this.productId = productId;
    this.name = name;
  }

  // Line 2: Override equals() to check only for productId equality
  @Override
  public boolean equals(Object o) {
    if (this == o) return true; // Same object reference
    if (o == null || getClass() != o.getClass()) return false;
    
    Product product = (Product) o;
    // Line 3: Two products are equal if their IDs match
    return productId == product.productId; 
  }

  // Line 4: Override hashCode() using only the productId
  @Override
  public int hashCode() {
    return Objects.hash(productId);
  }

  // Optional: Override toString() for better readability
  @Override
  public String toString() {
    return "Product ID: " + productId + ", Name: " + name;
  }
}

Line-by-Line Explanation

  • `public class Product { ... }` : Defines the class containing the data defining equality (`productId`).
  • `@Override public boolean equals(Object o) { ... }` : Defines the custom rule for equality, replacing the default memory address check.
  • `return productId == product.productId;` : The custom equality rule: only compare the ID, ignoring the `name` field.
  • `@Override public int hashCode() { ... }` : Returns a hash code generated *only* from the fields used in `equals()` (`productId`), maintaining the crucial contract.

Equals/HashCode Contract (Crucial Rule)

  • If `a.equals(b)` is **TRUE**, then `a.hashCode()` **MUST** equal `b.hashCode()`.
  • If you override `equals()`, you **MUST** override `hashCode()`, and vice versa.

Summary & Best Practices

  • Design classes with single responsibility ‚Äî one clear purpose.
  • Prefer composition over inheritance for flexible designs.
  • Use encapsulation to validate and protect internal state.
  • Override equals/hashCode when objects are value-like and will be used in collections.
  • Favor interfaces for capability contracts and abstract classes for shared code.
